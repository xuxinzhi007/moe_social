# 个人开发可用性基线清单

目标：在个人开发模式下，确保每次迭代都能稳定使用核心功能，避免“功能看起来完成但实际不可用”的半成品发布。

## 1. 使用目标与适用范围

- 适用对象：单人开发者、本地自测、预发布自检。
- 使用时机：每次准备发版、合并较大改动、跨模块重构后。
- 基线原则：先保证“可启动、可登录、可走完核心链路”，再继续新增功能。
- 验收要求：每条核心链路必须满足“可观察 + 可操作 + 有结果”三项标准。

## 2. 每次发版必须通过的核心链路与通过标准

以下链路全部通过，才允许发版。

### 2.1 启动

- 进入 App 后 10 秒内出现首个可交互页面（启动页/登录页/首页）。
- 不出现白屏卡死、持续 loading、明显报错弹窗。
- 可观察：有页面渲染、有状态反馈（加载结束）。
- 可操作：可点击至少一个主操作按钮（如登录、进入首页）。

### 2.2 登录

- 输入有效账号后，点击登录可进入已登录态页面。
- 输入错误信息时有明确提示文案，不可静默失败。
- 可观察：登录中状态可见（按钮 loading 或禁用）。
- 可操作：登录失败后可立即修改输入并再次提交。

### 2.3 首页

- 首页核心模块可正常渲染（至少包含主内容区和基础导航）。
- 首屏滚动、点击主入口无崩溃。
- 可观察：首屏数据或占位态展示完整，无布局溢出。
- 可操作：至少能进入一个二级页面并返回首页。

### 2.4 发现

- 发现页可打开并展示列表/卡片内容。
- 刷新或切换分类后有反馈（刷新动画、内容变化或空态提示）。
- 可观察：空数据时有空态文案，不是空白页。
- 可操作：可点击至少一个内容项进入详情或后续流程。

### 2.5 消息

- 消息页可打开并展示会话列表或空态。
- 进入单个会话后可发送一条消息并看到结果反馈。
- 可观察：发送中/发送失败状态可区分。
- 可操作：失败消息可重试，重试动作有结果。

### 2.6 个人页

- 个人页可打开并展示基础用户信息与关键入口。
- 编辑或进入设置类入口后可返回，状态不丢失。
- 可观察：用户信息加载失败时有可读提示。
- 可操作：至少一个设置项可修改并立即生效或有明确保存反馈。

### 2.7 VIP 购买

- VIP 页面可打开，套餐信息可见，选择套餐后可发起购买。
- 价格、权益说明、按钮状态一致，不出现“可点但无响应”。
- 可观察：下单前后状态变化清晰（处理中、成功、失败）。
- 可操作：失败后可重新发起，不需要重启 App。

### 2.8 订单

- 订单确认页可正确展示订单关键信息（商品、金额、状态）。
- 支付结果返回后，订单状态可刷新到最新结果。
- 可观察：成功/失败/取消三种结果有区分提示。
- 可操作：支持返回上页、重新下单或查看订单记录。

## 3. 异常恢复标准

异常处理必须满足“可重试、可提示、可回退”：

- 失败后重试：
  - 网络超时、接口 5xx、瞬时失败必须提供重试入口（按钮或下拉刷新）。
  - 重试最多连续 3 次；仍失败时给出下一步建议（稍后再试/检查网络）。
- 提示文案：
  - 文案必须说明发生了什么 + 用户能做什么。
  - 示例：`加载失败，请检查网络后重试`、`登录已过期，请重新登录`。
  - 禁止仅显示 `Error`、`请求失败` 等无动作指引的文案。
- 回退路径：
  - 关键流程失败时，用户可回到上一步，不可被卡在无操作页面。
  - 关闭弹窗、返回按钮、回首页入口至少保留一个可用。
  - 订单/支付失败后必须能重新发起购买或退出流程。

## 4. 每周迭代节奏模板（目标 -> 实现 -> 自测 -> 复盘）

建议按周循环，单周只追一个主目标，避免并行过多导致不可控。

- 目标（周一）：
  - 定义本周唯一主目标（例如“打通 VIP 下单闭环”）。
  - 写清完成定义：要通过哪些核心链路、达到什么可见结果。
- 实现（周二到周四）：
  - 先打通主链路，再补优化项。
  - 每天结束前至少跑一次核心链路冒烟（启动、登录、首页）。
- 自测（周五）：
  - 按本文第 2 节逐条验证，记录失败点与复现步骤。
  - 修复阻塞发布问题，非阻塞问题移入下周待办。
- 复盘（周末或下周一）：
  - 记录本周“做成了什么、卡在哪里、下周如何避免再卡”。
  - 更新基线清单中不合理项，保持清单长期可执行。

## 5. 发布前检查清单（必须全部通过）

- [ ] App 冷启动可在 10 秒内进入可交互页面。
- [ ] 登录成功、登录失败两条路径都验证通过。
- [ ] 首页、发现、消息、个人页均可正常进入并返回。
- [ ] VIP 购买链路可走通，失败后可重试。
- [ ] 订单信息展示正确，支付结果状态可刷新。
- [ ] 关键失败场景都有明确提示文案（非技术黑话）。
- [ ] 所有失败页面均存在可用回退路径（返回/重试/回首页）。
- [ ] 核心页面无明显布局错乱、不可点击、卡死问题。
- [ ] 本周新增功能至少完成一次端到端手动回归。

---

执行建议：这份清单应作为每次发版前的固定 gate。若任一核心链路不通过，本次版本默认不发布。
