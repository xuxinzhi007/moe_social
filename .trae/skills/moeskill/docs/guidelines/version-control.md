# 版本控制规范

## 概述

版本控制是软件开发过程中的重要环节，有助于团队协作、代码管理和版本追踪。本指南将详细介绍Moe Social项目的版本控制规范。

## 版本控制工具

项目使用Git作为版本控制工具，主要托管在GitHub平台。

## 分支管理

### 1. 分支策略

项目采用Git Flow分支管理策略，主要分支包括：

- **main**：主分支，用于生产环境
- **develop**：开发分支，用于集成测试
- **feature/**：功能分支，用于开发新功能
- **hotfix/**：热修复分支，用于修复生产环境问题
- **release/**：发布分支，用于准备发布

### 2. 分支命名规范

- **功能分支**：`feature/功能名称`，如 `feature/user-profile`
- **热修复分支**：`hotfix/问题描述`，如 `hotfix/login-error`
- **发布分支**：`release/版本号`，如 `release/1.0.0`

### 3. 分支操作流程

#### 功能开发流程

1. **创建分支**：从develop分支创建feature分支
   ```bash
   git checkout develop
   git pull
   git checkout -b feature/feature-name
   ```

2. **开发功能**：在feature分支上开发功能

3. **提交代码**：定期提交代码
   ```bash
   git add .
   git commit -m "feat: 功能描述"
   ```

4. **推送分支**：将分支推送到远程仓库
   ```bash
   git push origin feature/feature-name
   ```

5. **创建PR**：创建Pull Request到develop分支

6. **代码审查**：团队成员审查代码

7. **合并分支**：审查通过后合并到develop分支

#### 热修复流程

1. **创建分支**：从main分支创建hotfix分支
   ```bash
   git checkout main
   git pull
   git checkout -b hotfix/hotfix-name
   ```

2. **修复问题**：在hotfix分支上修复问题

3. **提交代码**：提交修复代码
   ```bash
   git add .
   git commit -m "fix: 问题描述"
   ```

4. **推送分支**：将分支推送到远程仓库
   ```bash
   git push origin hotfix/hotfix-name
   ```

5. **创建PR**：创建Pull Request到main和develop分支

6. **代码审查**：团队成员审查代码

7. **合并分支**：审查通过后合并到main和develop分支

#### 发布流程

1. **创建分支**：从develop分支创建release分支
   ```bash
   git checkout develop
   git pull
   git checkout -b release/version-number
   ```

2. **准备发布**：在release分支上进行发布准备，如版本号更新

3. **提交代码**：提交发布准备代码
   ```bash
   git add .
   git commit -m "chore: 准备发布版本号"
   ```

4. **推送分支**：将分支推送到远程仓库
   ```bash
   git push origin release/version-number
   ```

5. **创建PR**：创建Pull Request到main分支

6. **代码审查**：团队成员审查代码

7. **合并分支**：审查通过后合并到main分支

8. **创建标签**：在main分支上创建版本标签
   ```bash
   git checkout main
   git pull
   git tag -a v1.0.0 -m "版本1.0.0"
   git push origin v1.0.0
   ```

9. **合并回develop**：将release分支合并回develop分支

## 提交规范

### 1. 提交信息格式

提交信息应遵循以下格式：

```
<类型>: <描述>

<详细说明>

<可选的脚注>
```

### 2. 提交类型

- **feat**：新功能
- **fix**：修复bug
- **docs**：文档更新
- **style**：代码风格调整
- **refactor**：代码重构
- **test**：测试代码
- **chore**：构建过程或辅助工具的变动

### 3. 提交信息示例

```
feat: 添加用户个人资料页面

- 实现用户个人资料页面
- 添加头像上传功能
- 实现个人信息编辑功能

Closes #123
```

### 4. 提交规范

- **简洁明了**：提交信息应简洁明了，描述清楚变更内容
- **使用英文**：提交信息使用英文
- **类型前缀**：使用适当的类型前缀
- **关联issue**：如果与issue相关，应在提交信息中关联
- **避免大提交**：避免一次提交过多变更，保持提交粒度合理

## 代码审查

### 1. 审查流程

1. **创建PR**：开发者创建Pull Request
2. **分配审查者**：分配代码审查者
3. **审查代码**：审查者审查代码
4. **提出意见**：审查者提出修改意见
5. **修改代码**：开发者根据意见修改代码
6. **重新审查**：审查者重新审查代码
7. **合并PR**：审查通过后合并PR

### 2. 审查标准

- **代码质量**：代码质量是否符合项目标准
- **功能实现**：功能是否正确实现
- **安全性**：是否存在安全问题
- **性能**：是否存在性能问题
- **测试**：是否有足够的测试覆盖
- **文档**：是否更新了相关文档

### 3. 审查工具

- **GitHub**：使用GitHub的Pull Request功能进行代码审查
- **代码审查工具**：使用代码审查工具，如CodeReview

## 版本号管理

### 1. 版本号格式

项目使用语义化版本号（Semantic Versioning）格式：

```
主版本号.次版本号.修订号
```

- **主版本号**：重大变更，不兼容的API变更
- **次版本号**：新功能，向后兼容
- **修订号**：bug修复，向后兼容

### 2. 版本号更新

- **功能开发**：完成新功能后，更新次版本号
- **bug修复**：修复bug后，更新修订号
- **重大变更**：进行不兼容的API变更时，更新主版本号

### 3. 版本标签

在发布时，在main分支上创建版本标签：

```bash
git tag -a v1.0.0 -m "版本1.0.0"
git push origin v1.0.0
```

## 最佳实践

### 1. 代码提交

- **频繁提交**：频繁提交代码，保持提交粒度合理
- **提交信息清晰**：使用清晰、准确的提交信息
- **避免提交无关文件**：不要提交无关文件，如编译产物、日志文件等
- **使用.gitignore**：使用.gitignore文件排除不需要版本控制的文件

### 2. 分支管理

- **及时合并**：功能开发完成后及时合并到develop分支
- **定期清理**：定期清理不再需要的分支
- **保持分支同步**：定期从主分支同步更新

### 3. 代码审查

- **及时审查**：及时审查代码，避免阻塞开发进度
- **提供具体意见**：提供具体、 constructive的审查意见
- **尊重作者**：尊重代码作者，保持专业的沟通态度

### 4. 版本管理

- **版本号规范**：严格按照语义化版本号规范管理版本
- **发布流程**：遵循标准的发布流程
- **版本标签**：为每个发布版本创建标签

## 常见问题

### 1. 合并冲突

- **原因**：多个分支修改了同一文件的同一部分
- **解决方案**：手动解决冲突，然后提交

### 2. 分支混乱

- **原因**：分支管理不当，创建了过多分支
- **解决方案**：定期清理分支，保持分支结构清晰

### 3. 提交信息不规范

- **原因**：开发者没有遵循提交规范
- **解决方案**：制定提交规范，使用工具检查提交信息

### 4. 版本号管理混乱

- **原因**：版本号更新不规范
- **解决方案**：制定版本号管理规范，严格按照语义化版本号更新

## 总结

版本控制是Moe Social项目开发过程中的重要环节，良好的版本控制实践有助于提高开发效率、减少错误、确保代码质量。本指南提供了详细的版本控制规范，希望团队成员能够严格遵守，共同维护高质量的代码库。

在实际开发中，应不断总结和改进版本控制实践，以适应项目的发展和变化。
